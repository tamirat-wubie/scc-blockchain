# Pre-v0.8.4 Implementation Discovery — PATCH_05 §25 typed ProposalKind never wired

**Date:** 2026-04-20
**Class:** inter-session structural discovery (not a DCA pass against a specific draft — a discovery of an unresolved gap between spec and code that predates this session)
**Auditor:** drafter, during v0.8.4 implementation planning
**Filed under:** `docs/audits/` per the DCA-before-merge archival precedent (PATCH_10 §40.2 rule 2), because this finding directly gates PATCH_10 §38 implementation.

---

## Finding

While planning the implementation of PATCH_10 §38 (symmetric ceiling/param governance check), I searched for the `ModifyConsensusParam` `ProposalKind` variant that PATCH_05 §25 specifies. It does not exist.

Evidence:

- `crates/sccgub-governance/src/proposals.rs:40–56` — `ProposalKind` enum contains only:
  - `AddNorm`
  - `DeactivateNorm`
  - `ModifyParameter { key: String, value: String }` (the string-based path §25 was supposed to supersede for consensus params)
  - `ActivateEmergency`
  - `DeactivateEmergency`
- `crates/sccgub-types/src/typed_params.rs` — `ConsensusParamField`, `ConsensusParamValue`, and `apply_typed_param` DO exist, covering 30 of 31 `ConsensusParams` fields (I extended to all 31 in v0.8.3 via FRACTURE-V083-01 remediation).
- `crates/sccgub-governance/src/patch_04.rs:50–57` — `validate_consensus_params_proposal` DOES exist, accepting a pre-constructed `ConsensusParams` + `ConstitutionalCeilings`.
- **Callers of `validate_consensus_params_proposal` in production code: zero.** Only test-module + `crates/sccgub-node/tests/patch_04_conformance.rs` conformance test call it.
- `validate_ceilings_immutable` at `patch_04.rs:62–89` has the same property: declared, test-covered, but not wired into any submission path.

## Why this went unnoticed

1. `typed_params.rs` has its own unit tests that exercise `apply_typed_param` and the validators. These tests give the appearance of §25 being implemented.
2. The conformance test at `patch_04_conformance.rs:117–125` invokes `validate_consensus_params_proposal` directly, as if to simulate a production call. But the "production call" never happens because the `ProposalKind` variant doesn't exist.
3. `sccgub-audit` (the moat verifier) checks `ceilings_unchanged_since_genesis` — which is a read-side invariant. It does not check "is there a code path that could mutate ceilings?" The verifier reports the moat as HELD because in practice ceilings are never mutated; but the reason is "no code path exists," not "the governance path rejects mutations."

## Implication for PATCH_10

### §38 as-drafted is not directly implementable

PATCH_10 §38.2 describes rejecting proposals that would cause pair-invariant violation at activation. But there's no current proposal path that can mutate the `(ConstitutionalCeilings, ConsensusParams)` pair in a way that could violate the invariant:

- Ceilings: no variant exists to mutate them.
- Params: only string-based `ModifyParameter { key, value }` exists, and its handler's allowlist (I did not audit this live in this note — referenced from `STATUS.md` governance surface description) does not include any `ConsensusParams` field.

### §38 was based on a misreading of §17.8

The v0.5.0 DCA audit (§G.3, from 2026-04-18) said:

> **Ceiling-lowering attack.** Governance proposal to *lower* a ConstitutionalCeiling — not forbidden by §17.8 (which forbids raising only).

But the actual PATCH_04.md §17.8 text is:

> A governance proposal whose payload would modify any field in `ConstitutionalCeilings`, or whose modification of `ConsensusParams` would cause any pair in §17.2 to exceed its ceiling, is **rejected at submission**.

"Modify any field" rejects ceiling modifications **in either direction** (raising or lowering). The audit's claim that §17.8 "forbids raising only" was an error.

The real v0.8.3 state is stronger than §17.8 Rev-1 suggests: ceiling modification is impossible because no `ProposalKind` variant can carry it, AND `validate_ceilings_immutable` would reject it if one existed.

### Implication for #55

Issue #55 claims the ceiling-lowering attack is real under v0.5.0 semantics. Investigation reveals it is not: the attack requires a proposal variant that does not exist, and even the string-based `ModifyParameter` path has no handler that would write to `system/constitutional_ceilings`. The attack is structurally infeasible.

## Options for v0.8.4

**(A) Minimal defensive.** Wire `validate_ceilings_immutable` into the `ModifyParameter` submission path explicitly, rejecting any `key` that names a ceiling field. Amend PATCH_10 §38 to document the discovery. Close #55 as already-closed. ~100 LOC total.

**(B) Close the §25 gap (recommended).** Add `ProposalKind::ModifyConsensusParam { field: ConsensusParamField, new_value: ConsensusParamValue, activation_height: BlockHeight }`. Wire it through the submission pipeline. Calls `validate_consensus_params_proposal` at submission. Makes PATCH_10 §38 meaningful. ~400 LOC + conformance.

**(C) Withdraw §38.** Amend PATCH_10 to drop §38. Close #55 explicitly as wontfix-already-closed. Smallest honest move. ~50 LOC (spec amendment only).

## Recommendation

Option (B). Reasons:

1. Closes a real pre-session implementation gap (PATCH_05 §25 partially implemented since v0.4.0+, now ~12 months incomplete).
2. Makes PATCH_10 §38 meaningful — the symmetric check becomes an actual runtime protection rather than a description of code paths that don't exist.
3. Strengthens the governance surface with compile-time type safety for param modifications.
4. Doesn't require retroactive spec amendment (PATCH_10 stays as-merged).
5. Pre-existing `typed_params.rs` infrastructure does 80% of the work — mostly wiring work in `proposals.rs`, not new spec semantics.

Option (A) is acceptable as a stepping-stone. Option (C) would be intellectually honest but lose real capability.

## Sequencing

**v0.8.4:** implement Option B — `ProposalKind::ModifyConsensusParam` variant + wiring + typed-submission tests. Scope-bounded to §25 closure; does not touch §38 semantic composition or §39 evidence-layer veto.

**v0.8.5:** compose §38 (via `validate_consensus_params_proposal` called from the v0.8.4 submission path) + §39 (evidence-layer `ForgeryVeto` admission with f+1 attestation + no self-attestation).

Each release gets its own DCA pre-merge review per §40 discipline.
