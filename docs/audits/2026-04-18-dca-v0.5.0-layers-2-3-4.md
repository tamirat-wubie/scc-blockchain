# Deterministic Causal Audit — SCCGUB v0.5.0 (Layers 2–3–4)

**Date:** 2026-04-18
**Target tag:** `v0.5.0`
**Frozen spec surface:** PROTOCOL.md §1–§14 + PATCH_04.md §15–§19 + PATCH_05.md §20–§29
**Scope:** NEW fractures at Layer 2 (adversarial robustness), Layer 3 (regulatory / compliance), and Layer 4 (operational hardening) introduced or exposed by the Patch-04/05 amendments. Layer 1 F1–F6 treated as closed per spec.
**Method:** Adversarial environment assumed. Wall-clock noise, NTP spoofing, cartel formation, distributional shift, jurisdictional action, hardware constraint all treated as present.
**Tone:** Neutral. Institutional. Evidence-driven. Diagnostic, not constructive.

## Preamble: what this document is and is not

This is a **record artifact**, not a spec change. It archives the findings of a Deterministic Causal Auditor (DCA) pass conducted against the v0.5.0 tag after the Layer 1 fractures F1–F6 were reported closed by Patch-04 and Patch-05. It records the state of the system as of v0.5.0 from an adversarial structural perspective.

The document is preserved for three purposes:

1. **Accountability.** A third party inspecting the repository should be able to see what was known, when it was known, and how it was addressed. The audit is part of the record of how the chain evolved through v0.5.0 → v0.8.1.
2. **Template.** Future patches should be reviewed under the same adversarial discipline. This audit is the first formal instance of that discipline applied to a post-Layer-1 state; it is the reference format.
3. **Institutional memory.** Specific findings may be addressed by later patches; some may remain open. The archival preserves the original findings so that remediation decisions are traceable.

It is **not** a remediation plan. DCA produces diagnoses, not decisions. Decisions are governance-class work.

It is **not** a critique of Patch-04 or Patch-05. Both patches closed real Layer 1 fractures. The finding is that Layer 1 closure introduced coupling between mechanisms that created NEW fractures at Layers 2–4 — this is the nature of patch work, not a defect in it.

## A) Structural Weakness Summary

The v0.4.0–v0.5.0 amendments close six foundational fractures but introduce new structural coupling between mechanisms that were specified in isolation. Five categories of NEW weakness emerge:

1. **Timing-mismatch fractures** between veto windows, activation delays, and governance timelocks (§15.7 ↔ §12 ↔ §24).
2. **Unbounded state growth** from append-only registries (§18.3, §27) with no specified pruning, on an in-memory trie (§8), with no snapshot/fast-sync (explicitly deferred §28).
3. **Circular dependency** between validator set and fork choice, created by §15 replacing the implicit genesis-bound validator set with a mutable one while fork-choice remains unspecified.
4. **Economic-capture surface shift** from single-block producer manipulation (closed by §20) to rare-but-predictable 4-of-N leadership windows (opened by §20 + §16.2 deterministic leader schedule).
5. **Regulatory inversion:** hardened replay determinism (§20.4, §23, §26) plus hardened history retention (§27) now makes erasure, sanction enforcement, and redaction structurally impossible — the chain is *less* operable in regulated jurisdictions than v0.3.0 was.

Layer 1 is closed. Layer 2–4 are not. The v0.5.0 posture is "cryptographically harder, operationally fragile, regulatorily exposed."

## B) Invariant Failures

### B.1 Declared invariants that are conditionally enforceable

| ID | Statement | Condition under which enforcement fails |
|---|---|---|
| INV-VALIDATOR-KEY-COHERENCE | record-level `validator_id` == agent-level `active_public_key(H)` | Ambiguous in the transitional window `[rotation_height, effective_height)` — §18.6 permits either key. An attacker who compromises `old_validator_id` can sign conflicting votes during this window indistinguishable from legitimate transition traffic. INV admits both keys — so equivocation under the old key is not detectable as equivocation. |
| INV-CEILING-PRESERVATION | `param_value <= ceiling_value` at every height | Enforced at Phase 10 per-block. No invariant prohibits governance from **lowering** a ceiling in a way that invalidates currently-valid `ConsensusParams`. No invariant defines behavior if ceiling-lower proposal passes timelock while params are in-flight above new ceiling. Replay from genesis would reject all post-lowering blocks silently. |
| INV-HISTORY-COMPLETENESS | every admitted change appears in `system/validator_set_change_history` | Append-only with no pruning (§27.3) + unbounded churn rate (8 changes/block) = violates implicit invariant "state fits in memory on conforming hardware" (§8: "in-memory key-value trie"). Conformance at one year is incompatible with conformance at ten years on the same hardware profile. |
| INV-SEAL-NO-GRIND | registration seal matches `seal_v4(H, prior_block_hash)` | Only covers **registration seals** (§21.2). Header seals remain `from_height(height)` — pre-Patch-04 grinding surface on headers is intact. INV does not cover header seals even in v4. If any semantic weight is ever attached to header seals in later patches, F6 is reopened without spec change. |
| INV-FEE-ORACLE-BOUNDED | `T_median(H) ∈ [min(T_window), max(T_window)]` | Mathematically true. Does not bound `T_median` against *manipulation*. A 4-of-N cartel controlling ⌈W/2⌉ consecutive leaders can drive the median to any value in their chosen range. The invariant bounds arithmetic, not economic capture. |

### B.2 Missing invariants required for safety

| Missing invariant | Why critical |
|---|---|
| INV-FORK-CHOICE-DETERMINISTIC | §16 specifies round advancement but not fork-choice among competing chain heads. Post-partition reconciliation is unspecified. Two honest operators rejoining the network after partition may replay to different tips. |
| INV-STATE-BOUNDED | No invariant binds state-size growth. §27, §18.3 unconditionally append. Commodity-hardware operability has no upper time horizon. |
| INV-EVIDENCE-INCENTIVE | §22 admits evidence but no rule requires or rewards its submission. Evidence-sourced slashing is economically inert — no rational validator pays gas to attack a peer that retains voting power. |
| INV-VETO-FEASIBLE | §15.7 forgery-veto window is k+8 ≤ 10 blocks, but Safety-level governance timelock (§12) is 200 blocks. The veto is un-invokable through normal governance. |
| INV-KEY-REVOCATION-PATH | §18 KeyRotation requires signature by old key. Compromised-key recovery for non-validators has no specified path. |

## C) Assumption Map

### C.1 VERIFIED (evidence in spec)

- Canonical bincode encoding produces byte-identical output on conforming libraries (§1 + sccgub-types tests `patch_04_*_canonical_bytes`).
- `verify_strict` rejects malleable signatures (§26; ed25519-dalek documented behavior).
- Leader selection is replay-deterministic (§16.2 + `patch_04_leader_selection_deterministic`).

### C.2 PLAUSIBLE (reasonable, unproven)

- Cartel formation requires ⌈W/2⌉ consecutive leaders (§20.5) → probabilistic bound is correct *for uniformly-random leader draws*. Actual leader selection is deterministic, so the claim is really "probability that a random assignment of 4 attacker validators out of 64 yields 4-in-a-row in any W-block window somewhere along the chain." Geometric waiting time is low (≈65k blocks at n=64, f=4). Plausible but economically bounded, not cryptographically bounded.
- Validator set size ≤ 128 is "sufficient for BFT" (§15.8). Unproven against Byzantine-CAP performance under partition.

### C.3 FRAGILE (likely to fail under stress)

- "Ed25519 signers never produce non-canonical signatures" (§26.2). This is library-dependent, not protocol-enforced. A conforming *validator* could use a non-conforming signing library and produce verify-but-not-verify_strict signatures. §15.7 veto exists to mitigate this, but is structurally un-invokable (see B.2).
- "In-memory key-value trie" (§8) assumes state fits in RAM. False at scale.
- "Attacker pays for every miss" (§21.3). True only if base_fee × gas_price > 0 at registration time. Genesis and early-chain base_fee is near-zero.
- Local wall-clock divergence is non-consensus (§16.5). True for state-root output; false for *liveness* — divergent clocks produce divergent NewRound broadcasts, creating false view-changes under NTP-spoofing.

### C.4 CRITICAL (if false, system collapses)

- **Assumption A1:** State fits in memory. Falsified by §27 + §18.3 growth within 1–2 years.
- **Assumption A2:** Validators have access to honest NTP. Falsified by standard adversarial network model. §16.1 timeouts depend on it.
- **Assumption A3:** Governance can meaningfully exercise §15.7 veto. Falsified by §12 timelock mismatch.
- **Assumption A4:** All conforming nodes converge on the same fork after partition. Unspecified; no invariant enforces it.
- **Assumption A5:** Equivocation evidence will be submitted by someone. No economic incentive specified; rational submission rate is zero.

## D) Scaling Collapse Points

### D.1 Worst-case state growth

```
State growth per block =
    O(equivocation_evidence_per_block × 500 B)
  + O(validator_set_changes_per_block × 500 B)
  + O(key_rotations_per_block × 200 B)
  + O(transitions × avg_payload_bytes)
```

At defaults (4 evidence, 4 VSC, typical churn), ≈ 5 KB/block amortized *excluding* transitions.
At ceilings (16 evidence, 8 VSC), ≈ 12 KB/block.
At 1-second blocks × 3 years = ~94M blocks → 470 GB to 1.1 TB of validator-set + key-index state alone.

First collapse point: node bootstrap time exceeds block time. Once replay-from-genesis takes longer than `current_height × block_time`, new validators cannot catch up. Occurs at approximately year 2 on commodity hardware.

### D.2 Worst-case consensus message volume

- Per block at round R: up to `|active_set(H)|` prevotes + precommits + NewRound at each round 0..R.
- Ceiling: 128 validators × 3 message types × arbitrary R (no specified cap on R).
- `T_round` caps individual round latency but not round count. §16.1 "max timeout 60 min" with exponential backoff → round R takes at most `min(1s × 2^R, 60m)` — practically bounded to ≈ R=12 rounds per height.
- 128 × 3 × 12 = 4608 messages per block in the worst case. Byte-size: 4608 × 200 B ≈ 900 KB consensus traffic. `max_block_bytes_ceiling` = 8 MiB (§17.2) accommodates, but round_history commits every round's message set into the block, which **multiplies block size by round count**. A single high-round block can exhaust the 8 MiB ceiling and be rejected under §17.5, producing a liveness failure triggered by liveness failure — the feedback loop is unstable.

### D.3 First inevitable bottleneck

Node bootstrap time. No snapshot/fast-sync (§28 defers). Genesis-replay is the sole trust anchor (PATCH_04 §F8 deferral). Year 2 is the first inevitable bottleneck on commodity hardware.

### D.4 Governance-queue collapse

`max_active_proposals_ceiling=256` × 200-block constitutional timelock. An attacker controlling 1 validator can submit governance proposals at block-production rate. At 1 proposal/block × 256 slots = queue fills in 256 blocks. Every subsequent submission is rejected with `ProposalQueueFull`. Legitimate governance is blocked for 200 blocks (until first timelock expiry clears a slot). Effective governance bandwidth ≈ 1 proposal per 200 blocks under adversarial load.

## G) Adversarial Surface

### G.1 Input spoofing paths

- **Network re-org harvesting equivocation evidence** against honest validators who voted on two forks (§15.7 definition of equivocation does not distinguish re-org-induced double-voting from malicious double-voting). Path: attacker triggers shallow re-org; harvests evidence for every validator who voted both sides; submits up to 16 evidence records/block; triggers mass synthetic Remove. Veto window (10 blocks) < Safety-level governance timelock (200 blocks) → no defense.
- **Non-canonical signature construction** to exploit §15.7 veto as a DoS. Attacker constructs signatures that `verify` accepts but `verify_strict` rejects. Submits as "forgery proof" via Safety-level proposal to veto legitimate equivocation removal. Each proposal occupies a queue slot for 200 blocks.
- **NTP-spoof-induced view change cascade.** An attacker who can influence a single validator's NTP source drives that validator to issue premature `NewRound` broadcasts, creating phantom view changes that burn block space via `round_history` (§16.6).

### G.2 Resource exhaustion vectors

- **State growth attack** via high-frequency `ValidatorSetChange::RotatePower` (§15.4) with +1/-1 voting power oscillation. Each change appends to `system/validator_set_change_history` (§27). No rate limit per agent_id; only `max_validator_set_changes_per_block=8` limits *submission* rate, not cumulative growth.
- **Key-index pollution** via registration-then-abandonment of distinct keys. KeyIndex (§18.3) is permanent, sorted by public_key. An attacker with cheap registration (low base_fee early chain) registers and rotates through millions of keys; KeyIndex grows linearly.
- **Governance queue fill** — 256 slots × 200-block timelock = 200-block legitimate-governance blackout for cost of 1-block producer control.
- **round_history bloat** — attacker in leader position declines to prevote, forces round advancement; round_history grows; block size approaches `max_block_bytes_ceiling`; legitimate txs displaced.

### G.3 Rule poisoning paths

- **Ceiling-lowering attack.** Governance proposal to *lower* a ConstitutionalCeiling — not forbidden by §17.8 (which forbids raising only). Lowering a ceiling below current `ConsensusParams` value invalidates current state under INV-CEILING-PRESERVATION at the activation height. Replay after activation fails for all blocks between proposal and current tip.
- **Ceiling → parameter drift exploit.** §17.2 ceiling table includes 13 pairs; §29 adds 4 more. The relationship "param ≤ ceiling" is enforced per-block but no invariant enforces the *spread* (param/ceiling ratio). Governance can raise a param to 99% of ceiling legitimately, then future ceiling adjustments have no headroom.

### G.4 Privilege escalation paths

- **Validator key rotation under quorum collusion.** §18.6 requires quorum signatures. A 2/3+1 quorum can unilaterally rotate any validator's key to a key of their choosing (variant predicate §15.5.1 permits `new_validator_id not in use by any other ValidatorRecord`, which is satisfiable with a fresh attacker key). The victim validator is locked out of their own agent_id while retaining it nominally. This is a **validator-set capture path** that Patch-04 §18.6 closed for the compromised-key threat model but opened for the compromised-quorum threat model.
- **Slashing as privilege attack.** 2/3+1 quorum can submit EquivocationEvidence against any validator by constructing the two conflicting votes themselves (sign as the target in a forked-block scenario the target never saw). §22 admission requires signatures verify and validator was in active set — both trivially true for target. Target is slashed; quorum retains control.

### G.5 Safety bypass attempts

- **Soft-finality lie.** §7 "Soft (depth 0)" finality class is immediate. An application relying on soft finality for a high-value transfer is exposed to re-org rollback. Nothing in protocol prevents a 0-confirmation transfer that is later invalidated. This is a misuse failure mode, not a protocol failure, but PROTOCOL.md does not label soft-finality as unsafe for value above a threshold.
- **Transitional-key window abuse.** §18.6 permits EITHER old or new validator_id during `[rotation_height, effective_height)`. An attacker who compromises `old_validator_id` and also obtains `new_validator_id` can double-sign conflicting votes under the two keys with plausible deniability (vote under old key was "clock skew", vote under new key was "legitimate transition").

### G.6 Fleet desynchronization exploits

- **Fork-choice divergence.** No canonical fork-choice rule specified. Nodes that observe two chain heads must choose; no deterministic rule. Byzantine validator can publish two blocks at the same height with valid signatures and different transaction sets — honest nodes diverge.
- **Post-partition reconciliation.** §19.5 / §28.4 says "no in-place upgrade." No clause addresses post-partition reconciliation. Two sub-networks that forked admit divergent `ValidatorSetChange` histories; rejoining produces two inconsistent `system/validator_set_change_history` entries — §27.4 INV-HISTORY-COMPLETENESS is violated by either branch accepting the other's additions.

## H) Fracture Ranking — Top 5

> **Note on numbering:** this audit's H.1–H.5 uses the standard DCA output-format numbering (structural-fracture rank, ordered by combined severity × detectability × containment). This numbering is *independent* of any other H.1–H.5 used by later patches in this repository (e.g., PATCH_06.md uses its own H.1–H.5 enumeration scoped to what that patch closes). Readers comparing the two should treat them as separate namespaces. The cross-map appears in the "Post-hoc remediation status" section below.

### H.1 FRACTURE-L2-01: Veto–Timelock Structural Mismatch (§15.7 ↔ §12)

- **Trigger:** Any EquivocationEvidence admission that an honest validator wishes to contest via forgery proof.
- **Cascade:** Safety-level governance timelock (200 blocks) > activation_delay (≤10 blocks). Veto must be admitted AND activated within activation_delay, but governance cannot activate anything in < 50 blocks. Honest validator is slashed before veto is activatable.
- **Detectability:** Low — the mechanism appears specified. Only joint timing analysis surfaces it.
- **Containment:** None in current spec. Requires Patch-06 or greater to reconcile.

### H.2 FRACTURE-L4-02: Unbounded State Growth + No Fast-Sync

- **Trigger:** Normal chain operation over 12–24 months.
- **Cascade:** Node bootstrap time exceeds new-block production rate. New validators cannot join. Validator set ossifies toward existing members. Eventually the chain is unrunnable on commodity hardware.
- **Detectability:** High — measurable in production.
- **Containment:** Partial — operators can run larger hardware. Temporary relief, not structural fix. Requires state pruning (explicitly deferred §28) or statelessness (unspecified) to resolve.

### H.3 FRACTURE-L2-03: Quorum-Collusion Validator-Set Capture

- **Trigger:** 2/3+1 validator collusion (same threshold that is already assumed-honest in BFT).
- **Cascade:** Colluding quorum can key-rotate arbitrary non-colluding validators to attacker keys; can slash arbitrary validators via self-constructed equivocation evidence. Chain governance is effectively captured.
- **Detectability:** Medium — post-hoc forensic analysis can show the pattern but no per-block alert.
- **Containment:** None at protocol level. Mitigation is social (validator selection diligence).

### H.4 FRACTURE-L3-04: Regulatory Impossibility Lock-In

- **Trigger:** Any jurisdictional request for erasure, redaction, or sanction enforcement.
- **Cascade:** Protocol provides no path to comply. Operators face choice between non-compliance (legal risk) or forking (breaks INV-3). In EU or OFAC-reach jurisdictions, the chain cannot be operated compliantly.
- **Detectability:** High — first subpoena triggers it.
- **Containment:** Private-fork operation (loses consensus with public chain). No structural fix without Patch-N regulatory-aware redaction mechanism, which fundamentally conflicts with §13 replay determinism.

### H.5 FRACTURE-L2-05: Fork-Choice Circular Dependency

- **Trigger:** Any network partition lasting > confirmation_depth blocks with validator-set change during the partition.
- **Cascade:** Two sub-networks produce divergent `ValidatorSetChange` histories. Rejoining requires deciding which history is canonical; validator set itself depends on the choice. No deterministic resolution specified.
- **Detectability:** High — observable at re-org time.
- **Containment:** Manual — operator selects. Not consensus-compatible; different operators may select differently, fragmenting the network permanently.

## Reconciliation with "Layer 1 closed"

Layer 1 is substantially closed. F1–F6 map to mechanisms (§15, §16, §17, §18, §20, §21) that are spec-complete. Conformance matrix lists 37+ Patch-04 tests and 17 Patch-05 tests. The closure is real.

The audit finding is **not** that Layer 1 is unclosed. The finding is that Layer 1 closure **introduced** coupling between mechanisms (veto × timelock, history × memory, deterministic replay × regulatory erasure) that create NEW fractures at Layers 2–4. Five of the top fractures above did not exist in v0.3.0 because the corresponding mechanisms did not exist.

# Advisory note — DCA-before-merge discipline

This audit is the first formal adversarial structural pass conducted against SCCGUB after Layer 1 closure. It found coupling-induced fractures that approval-style review had missed. The finding has a direct implication for working discipline:

**From the next patch forward, no patch should merge until:**

1. An adversarial structural review has been run against the patch spec by an agent not involved in drafting the patch (tool-agnostic: the Deterministic Causal Auditor skill is one instance of this discipline; any equivalent adversarial framework satisfies the requirement);
2. The review's findings are attached to the PR as a record artifact (in `docs/audits/` alongside this file);
3. Each finding is either remediated in the same PR or explicitly deferred with a dated, labeled tracking issue.

This is a governance discipline change. Per POSITIONING §13 amendment discipline, it should be adopted via a documented amendment in the PR that first enforces it — i.e., the next specification patch should carry a §13 amendment instituting this rule. Informal conventions decay; institutional commitments require formal adoption.

## Register separation (for the record)

- **Audit produces findings.** Diagnostic. Neutral. "Neutral finding. No recommendations" is the correct close.
- **Advisory produces recommendations.** Tactical. Trade-offs explicit. Answers "what could be done."
- **Operator produces decisions.** Governance. Answers "what will be done."

The three roles must not collapse into one; if they do, the audit's value — adversarial distance from the thing being audited — is lost.
